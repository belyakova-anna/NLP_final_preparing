# Билет 19

## Q1. Model Context Protocol (MCP) for LLM agent tools

**Model Context Protocol (MCP)** — открытый протокол, представленный **Anthropic в ноябре 2024** для стандартизации того, как LLM-агенты подключаются к внешним инструментам, источникам данных и контексту. Цель MCP — устранить «N×M-проблему»: для интеграции $N$ LLM-приложений с $M$ внешними системами раньше нужно было $N \times M$ разных коннекторов; MCP даёт единый стандарт, и $N + M$ интеграций становится достаточно.

### Зачем нужен

До MCP у каждой агентной системы был свой формат tool-calling (OpenAI function calling, LangChain Tools, Claude Tool Use, кастомные плагины ChatGPT). Каждое приложение реализовывало свои коннекторы к Gmail, Slack, Google Drive, GitHub, базам данных и т. д. — в результате тратились ресурсы на одну и ту же интеграционную работу. MCP — это «USB-C для LLM»: один стандарт между LLM-клиентами и tool-серверами.

### Архитектура

MCP построен на модели **client-server** через JSON-RPC 2.0:

- **Host** — приложение, в котором живёт LLM (Claude Desktop, Cursor, IDE-плагин, Cowork, кастомный агент);
- **MCP Client** — модуль внутри host'а, который держит соединение с сервером и предоставляет LLM ручки;
- **MCP Server** — отдельный процесс, оборачивающий конкретный сервис (БД, файловую систему, GitHub API, Slack и т. д.) и публикующий его возможности по протоколу.

Один Host может одновременно говорить с несколькими MCP-серверами. Транспорт: **stdio** (для локальных серверов, запускаемых как процессы), **HTTP/SSE** (для удалённых серверов).

### Что предоставляет MCP-сервер

Сервер декларирует три типа сущностей:

1. **Tools (инструменты)** — функции, которые LLM может вызвать: `search_database(query)`, `send_email(to, subject, body)`, `create_pr(title, branch)`. Каждый tool описан JSON-Schema-ой параметров.
2. **Resources (ресурсы)** — данные, которые LLM может прочитать: файлы, документы, строки БД, страницы. Идентифицируются URI (например, `file:///project/README.md`, `slack://channel/general`).
3. **Prompts (промпты)** — параметризованные шаблоны, которые сервер предоставляет клиенту: например, «summarize this PR», «debug this stack trace».

Дополнительно сервер может декларировать **Sampling** (запросить генерацию у клиентского LLM — чтобы сервер тоже мог использовать LLM) и **Roots** (рабочие директории/контексты).

### Жизненный цикл соединения

1. **Initialization**: client и server обмениваются версиями протокола и capabilities;
2. **Discovery**: client запрашивает `tools/list`, `resources/list`, `prompts/list`;
3. **Operation**: LLM в host'е видит описания и при генерации может выпускать `tools/call`, `resources/read`, `prompts/get`;
4. **Notifications**: server может пушить изменения (например, файл обновился — `notifications/resources/updated`);
5. **Termination**: корректное закрытие сессии.

Все сообщения — JSON-RPC 2.0 с `method`, `params`, `id`. Например, вызов tool:

```json
{
  "jsonrpc": "2.0",
  "id": 42,
  "method": "tools/call",
  "params": {
    "name": "github_create_issue",
    "arguments": {"repo": "anthropic/mcp", "title": "..."}
  }
}
```

### Сравнение с function calling

- **Function calling** (OpenAI, Anthropic) — это формат вывода LLM (JSON с именем функции и аргументами). Что делать с этим вызовом — забота приложения, протокола между приложением и backend'ом нет.
- **MCP** — это стандарт для самого backend'а: как сервер описывает свои tools, как они обнаруживаются, как вызываются, как возвращаются результаты, как стримятся ресурсы.

То есть function calling и MCP — комплементарны: LLM генерирует function call в своём формате, host транслирует его в `tools/call` к нужному MCP-серверу.

### Преимущества

1. **Переносимость** — один и тот же сервер работает с Claude Desktop, Cursor, кастомными агентами;
2. **Композируемость** — host может подключить десяток серверов одновременно;
3. **Безопасность** — отдельный процесс/sandbox, контроль прав на уровне host'а;
4. **Экосистема** — растёт каталог open-source MCP-серверов (GitHub, Slack, Google Drive, Postgres, Filesystem, Puppeteer/Playwright и т. п.);
5. **Двусторонность** — server может запросить sampling у LLM клиента, что позволяет полу-агентные сценарии на стороне сервера.

### Где используется

- **Claude Desktop / Claude Code** — MCP — основной механизм расширений;
- **Cursor, Continue.dev, Zed** — поддержка как клиенты;
- **Cowork** (этот клиент) — плагины через MCP;
- **OpenAI** в 2025 объявила о поддержке MCP в Agents SDK и ChatGPT;
- **Open-source каталог** (Anthropic + сообщество) — сотни готовых серверов.

## Q2. PEFT. Low-Rank Adaptation (LoRA)

**PEFT (Parameter-Efficient Fine-Tuning)** — семейство методов дообучения больших моделей, при которых **обновляется лишь малое подмножество параметров** (или вводятся дополнительные малые параметры), а основные веса замораживаются. Мотивация: full fine-tuning 70B-модели требует сотен GB GPU-памяти и storage; PEFT даёт сравнимое качество при ~0.01–1% параметров.

### Семейство методов PEFT

1. **Adapter modules** (Houlsby et al., 2019) — между блоками трансформера вставляется маленький bottleneck-слой `down → nonlinearity → up` с residual. Обучаются только адаптеры.
2. **Prefix Tuning / Prompt Tuning** (Li & Liang, 2021; Lester et al., 2021) — обучаемые «виртуальные» токены добавляются в начало последовательности (или в каждом слое). Основная сеть заморожена.
3. **(IA)³** (Liu et al., 2022) — три обучаемых вектора $l_k, l_v, l_{ff}$ масштабируют ключи, значения и активации FFN. Очень мало параметров.
4. **BitFit** (Zaken et al., 2021) — обновляются только bias-параметры. Дёшево, но качество ниже LoRA.
5. **LoRA** (Hu et al., ICLR 2022) — главный сегодня стандарт, разбираем подробно.
6. **LoRA-варианты**: **DoRA** (декомпозиция magnitude/direction), **QLoRA** (LoRA + 4-битная база), **LoRA+** (разные lr для A и B), **AdaLoRA** (адаптивный ранг), **LoHA / LoKR**, **VeRA**.

### LoRA — Low-Rank Adaptation

**Идея**: при дообучении изменение веса $\Delta W$ для каждой матрицы ограничивается **низким рангом**. Вместо обучения полного $\Delta W \in \mathbb{R}^{d \times k}$ обучаются две маленькие матрицы $A \in \mathbb{R}^{r \times k}$ и $B \in \mathbb{R}^{d \times r}$, причём $r \ll \min(d, k)$:

$$W' = W_0 + \Delta W = W_0 + B A,$$

или с масштабированием:

$$h = W_0 x + \frac{\alpha}{r} B A x.$$

- $W_0$ — замороженная исходная матрица (не обучается);
- $A$ инициализируется $\mathcal{N}(0, \sigma^2)$;
- $B$ инициализируется **нулями** — в начале $\Delta W = 0$, модель начинает с базы;
- $r$ — ранг адаптера (обычно 4–64);
- $\alpha$ — масштабный коэффициент (типично $\alpha = r$ или $\alpha = 2r$).

### Куда применять LoRA

В трансформере LoRA применяют к **линейным проекциям**:
- $W_Q, W_V$ — стандартный набор (по оригинальной статье);
- $W_Q, W_K, W_V, W_O$ — расширенный;
- $W_{up}, W_{down}, W_{gate}$ FFN — даёт ещё прирост на сложных задачах;
- Embeddings и LM-head обычно не трогают.

Чем больше слоёв охвачено LoRA, тем выше качество и тем больше памяти/параметров. Стандартный recipe в LLaMA-fine-tuning — все 7 матриц (`q_proj, k_proj, v_proj, o_proj, gate_proj, up_proj, down_proj`).

### Преимущества

1. **Малое число параметров**: для LLaMA-7B и $r=8$ LoRA добавляет ~4M параметров (≈0.06% от общих), не считая базовой модели;
2. **Память**: оптимизатор хранит состояния (Adam) только для LoRA-параметров — гигантская экономия (Adam = 8 байт/параметр, поэтому full FT требует 16–24 GB памяти на градиенты+оптимизатор для 7B; LoRA — пренебрежимо);
3. **Безпотерь по инференсу**: можно **слить** LoRA в базовые веса: $W_{merged} = W_0 + BA$, и инференс идёт обычной матмул без overhead;
4. **Композируемость**: для одной базы можно держать много LoRA-чекпойнтов (по доменам/задачам) и переключать;
5. **Скорость обучения**: меньше параметров обновляется, активации и градиенты дешевле;
6. **Совместимость с квантизацией**: QLoRA — база в NF4, LoRA в FP16, fine-tune 65B на одной 48GB GPU.

### Недостатки

- На очень сложных задачах (новый домен, новый язык) full fine-tuning может выигрывать;
- Чувствительность к выбору $r$ и $\alpha$;
- Не всегда понятно, какие модули охватывать;
- LoRA для **embedding-слоя** работает плохо (там естественно нужен большой ранг);
- Multiple-task interference при одновременном использовании нескольких LoRA.

### QLoRA (важный частный случай)

**QLoRA** (Dettmers et al., 2023):
- Базовые веса хранятся в **NF4** (NormalFloat-4) — нелинейная 4-битная схема под нормально-распределённые веса;
- LoRA-адаптеры в FP16 / BF16;
- **Double quantization** — квантуются и сами scale-параметры;
- **Paged optimizers** — Adam-состояния выгружаются в CPU RAM при OOM.

Результат: fine-tune **65B**-модели на одной **48 GB GPU** с качеством, близким к full fine-tuning в FP16.

### Связь с другими темами

PEFT/LoRA — обязательная составляющая современного стека alignment: типичный pipeline это `pretrained LLM → SFT через LoRA → DPO через LoRA → merge или hot-swap`. Также часто применяется поверх квантизованной модели (QLoRA) для домен-адаптации без аренды дорогих GPU.
