# Билет 18

## Q1. Planning in agents

**Planning** в LLM-агенте — это процесс **декомпозиции цели в последовательность (или граф) шагов** перед или во время исполнения. Хорошее планирование критически важно для длинных, многошаговых задач: без него агент действует реактивно и накапливает ошибки.

### Зачем планирование

LLM сильна в локальных reasoning-шагах, но плохо удерживает когерентность на длинных горизонтах. Если агенту дать сложную цель типа «провести анализ продаж за квартал и подготовить презентацию для CEO», без явного плана он:
- забудет часть задачи;
- сделает шаги в неверном порядке;
- застрянет в локальных деталях.

Планирование решает три вещи: **декомпозиция**, **последовательность/зависимости**, **контроль прогресса**.

### Базовые стратегии планирования

#### 1. Plan-and-Execute (Sequential planning)

Wang et al., 2023. Двухфазная модель:

1. **Planner-LLM** строит линейный план по цели: `[step1, step2, ..., stepN]`;
2. **Executor-LLM** выполняет каждый шаг последовательно (часто в режиме ReAct), накопляя результаты.

Плюсы: простота, явность плана. Минусы: жёсткий план плохо адаптируется при ошибках в исполнении.

#### 2. ReAct (Reactive planning)

Yao et al., 2022. План **переплетён** с действиями: на каждом шаге Thought-Action-Observation. План формируется неявно, в процессе. Хорошо для коротких/средних задач, плохо для длинных (теряется глобальная стратегия).

#### 3. Hierarchical planning

Иерархия: high-level план → sub-plans → atomic actions. Пример: HuggingGPT — GPT-планировщик выбирает HuggingFace-модели, BabyAGI — task list с приоритизацией.

Плюсы: масштабируемо, переиспользуемо. Минусы: сложнее отладка.

#### 4. Tree-of-Thoughts (ToT)

Yao et al., NeurIPS 2023. Каждый «шаг рассуждения» — узел дерева, агент **разветвляет** возможные продолжения, эвристически оценивает их и идёт по лучшей ветке. По сути BFS / DFS / heuristic search в пространстве промежуточных мыслей.

$$\text{Score}(node) = \text{LLM-self-eval}(node) \quad \text{или RM-score}.$$

Полезно для задач, где нужно явно перебирать варианты (math puzzles, planning, creative writing).

#### 5. Graph-of-Thoughts (GoT)

Besta et al., 2023. Обобщение ToT: вершины-мысли, рёбра-зависимости, можно объединять подцели. Гибче, но дороже.

#### 6. Chain-of-Thought planning + Decomposition

Простой подход: prompt «list the steps needed to solve X» → LLM выдаёт пронумерованный план → исполнение каждого шага в отдельном промпте. Это «бедный» Plan-and-Execute, но часто работает.

#### 7. LLM Compiler (Kim et al., 2023)

Планировщик строит **DAG** из подзадач с зависимостями, исполнитель параллелит независимые ветви. Сильно ускоряет multi-tool агентов.

### Reflection / Self-critique / Replanning

Хорошие агенты не просто следуют плану, но и **переоценивают** его:

- **Reflexion** (Shinn et al., 2023) — после неудачной попытки агент анализирует, что пошло не так, формирует «урок» и сохраняет в памяти для следующей попытки;
- **Self-critique** — отдельный шаг или sub-agent, который оценивает каждый промежуточный результат, при failure триггерит replan;
- **Replanning** — изменение плана на лету при новых наблюдениях (например, tool вернул ошибку → пересмотреть стратегию);
- **CRITIC** (Gou et al., 2023) — критика и коррекция через внешние tools (поиск, код).

### Поиск в пространстве планов

Помимо одной траектории, можно искать оптимальный план:

- **MCTS-style search** — Tree-of-Thoughts с rollout'ами и upper-confidence bound;
- **Best-of-N** — генерируем N планов, оцениваем каждым (LLM-judge или verifier), выбираем лучший;
- **Beam-search planning** — beam над деревьями плана.

### LLM как world model для планирования

LLM-планировщик неявно использует LLM как **модель мира**: он симулирует, что произойдёт после действия. Это связь с темой «LLMs as world models» (билет 9). Чем точнее модель предсказывает последствия, тем лучше план.

В CodeAct (Wang et al., 2024) и AlphaCodium плановое поведение становится явным: агент сначала пишет «план» или «спецификацию», потом отдельно — код, потом тестирует.

### Известные agent-планировщики

- **AutoGPT, BabyAGI** (2023) — ранние OSS-агенты с явным task list;
- **HuggingGPT** — GPT-4 планирует pipeline из моделей HuggingFace;
- **Voyager** — GPT-4 планирует curriculum в Minecraft, накопляет skill library;
- **OpenAI Deep Research / o1 / o3** — встраивают «скрытое» планирование в reasoning chain;
- **Claude Computer Use, Devin (Cognition)** — длинноhorizon-агенты с явным планированием;
- **MetaGPT** — multi-agent framework с ролями (PM, architect, engineer, QA), каждый строит свой план.

### Метрики и бенчмарки планирования

- **PlanBench** (Valmeekam et al., 2023) — формальные планнинг-задачи, оценка LLM как планировщика. Результат: даже GPT-4 справляется ограниченно;
- **AgentBench, GAIA, SWE-bench, WebArena** — multi-step задачи, требующие планирования;
- **Blocksworld** — классическая планировочная задача из AI.

### Ограничения LLM-планирования

- **Long-horizon failure** — на 30+ шагах ошибки накапливаются;
- **Plan staleness** — план устаревает при изменении состояния среды, нужен replan;
- **Ложная уверенность** — LLM строит правдоподобный план, который физически невозможен;
- **Недостаточная верификация** — без внешнего verifier'а ошибки не отлавливаются;
- **Cost** — много токенов на промежуточные размышления.

Современная практика: гибридные подходы — высокоуровневый план Plan-and-Execute + локальный ReAct внутри каждого шага + reflection поверх + verifier (LLM-judge или внешние тесты).

## Q2. KNN-LM algorithm

**KNN-LM** (Khandelwal et al., ICLR 2020, «Generalization through Memorization: Nearest Neighbor Language Models») — token-level retrieval, дополняющий уже обученную LM на inference-time через k-ближайших соседей по эмбеддингам контекстов, **без какого-либо дополнительного обучения**. Канонический пример непараметрической памяти: LM знает язык, datastore хранит факты.

### Три фазы алгоритма

#### Phase 1 — Build Datastore

По всему обучающему корпусу $D$ строится datastore пар (key, value):

$$(K, V) = \{(f(c_i), w_i)\,|\,(c_i, w_i) \in D\},$$

где:
- $c_i$ — leftward-контекст (предыдущие токены до позиции $i$);
- $w_i$ — целевой токен в позиции $i$;
- $f(\cdot)$ — функция кодирования контекста, как правило hidden state из последнего слоя предобученного Transformer-декодера прямо перед softmax.

Индексация делается через **FAISS** для быстрого ANN-поиска. Размер datastore — миллиарды записей (по записи на каждый токен корпуса).

#### Phase 2 — Search KNN contexts

На inference, для текущего тестового контекста $x$ вычисляется query-эмбеддинг $q = f(x)$, и в FAISS ищутся $k$ ближайших ключей по евклидовой или косинусной дистанции:

$$d_i = d(q, k_i), \quad i = 1, \ldots, k.$$

Получаем $k$ соседей $\{(k_i, v_i)\}$.

#### Phase 3 — Aggregation and Interpolation

Расстояния превращаются в распределение через softmax с обратным знаком:

$$p(k_i) \propto \exp(-d_i),$$

затем агрегируются по целевым токенам:

$$p_{\text{kNN}}(y \mid x) = \sum_{(k_i, v_i) \in \mathcal{N}}\, \mathbb{1}_{y = v_i}\, p(k_i).$$

Несколько соседей с одним и тем же target-токеном складывают свои массы. Например, если 3 соседа возвращают «Hawaii» — их вероятности суммируются (типичный пример в иллюстрациях из статьи: ~0.8 для «Hawaii»).

Итоговое распределение — **линейная интерполяция** kNN-распределения и распределения исходной LM:

$$p(y \mid x) = \lambda\, p_{\text{kNN}}(y \mid x) + (1 - \lambda)\, p_{\text{LM}}(y \mid x),$$

где $\lambda \in [0, 1]$ — гиперпараметр (обычно подбирается на валидации, типичные значения 0.1–0.3).

### Результаты и инсайты

- На **WikiText-103 perplexity падает с 18.65 до 15.79** (~16% относительного улучшения);
- **Дополнительного обучения LM не требуется — ZERO**;
- Качество **масштабируется с размером datastore**: модель, обученная на 100M токенов + datastore на 1.6B токенов, обходит ту же модель, обученную на 3B токенов с нуля;
- Особенно полезен для **long-tail / rare**-паттернов: фактических знаний, именованных сущностей, доменных терминов — то есть именно там, где параметрическая память галлюцинирует;
- **Zero-shot domain adaptation**: можно подменить datastore на медицинский или юридический корпус, и модель мгновенно адаптируется без fine-tuning.

### Как именно помогает rare patterns

Параметрическая память LM «размазана» по весам и хорошо запоминает частотные паттерны, но плохо — редкие имена, цифры, специфические факты. KNN-LM делает явный поиск по точным контекстам — для редкого паттерна найдётся точный сосед, и его вероятность будет переоценена через kNN-член.

### Сравнение KNN-LM / RAG / RETRO

| Свойство | KNN-LM | RAG | RETRO |
|---|---|---|---|
| Unit retrieval'а | токен | document/passage | chunk (64 tok) |
| Этап | inference-time | inference-time | pre-training |
| Интеграция | вероятностная интерполяция $\lambda p_{kNN} + (1-\lambda) p_{LM}$ | конкатенация в промпт | chunked cross-attention |
| Архитектура | без изменений | без изменений | новые CCA-слои |
| Масштаб БД | миллиарды | миллионы | триллионы |
| Доп. обучение | нет | опционально | да (с нуля или RETROfit) |

### Минусы и ограничения

- **Latency** — на каждом шаге генерации нужен FAISS-поиск (хотя ANN быстрый);
- **Память** — datastore на миллиарды эмбеддингов размером $d \approx 1024$ — это десятки/сотни GB;
- **Качество $f$** — если эмбеддинги контекста плохо отделяют семантику, kNN не найдёт хороших соседей;
- **Генерация креативного текста** — kNN тянет к виденным паттернам, может снижать diversity.

### Развитие

- **Adaptive kNN-LM** — учить $\lambda$ адаптивно по контексту;
- **kNN-MT** — то же для машинного перевода;
- **Datastore distillation** — сжатие миллиардов записей в меньший индекс.
