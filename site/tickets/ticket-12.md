# Билет 12

## Q1. Definition of an LLM agent

**LLM-агент** — это автономная система, в которой **большая языковая модель используется как ядро принятия решений**, способная воспринимать окружение, планировать, выполнять действия (в т. ч. через внешние инструменты) и итеративно корректировать поведение для достижения поставленной цели.

### Формальное определение

Агент — это сущность, которая в **цикле**:

$$\text{percept} \xrightarrow{\text{LLM-decision}} \text{action} \xrightarrow{\text{environment}} \text{observation} \xrightarrow{} \text{percept}'$$

взаимодействует с окружением. У классического агента (Russell & Norvig) есть: sensors → percepts → reasoning → actuators → actions. В LLM-агенте:
- **sensors / percepts** — текстовый ввод (запрос пользователя, наблюдения от tools, retrieved-документы);
- **reasoning** — LLM, генерирующая мысли и решения о следующем действии;
- **actuators / actions** — вызовы tools, генерация ответа, запись в память;
- **environment** — пользователь, файловая система, интернет, IDE, БД, API, другие агенты.

### Ключевые характеристики LLM-агента

1. **Autonomy (автономность)** — агент сам определяет последовательность шагов. Получив высокоуровневую цель («собери отчёт по продажам за Q1»), он сам выбирает источники, инструменты, формат и порядок действий.
2. **Goal-directedness** — поведение направлено на достижение конкретной цели, а не на выдачу одного ответа.
3. **Reactivity** — реакция на наблюдения от среды (tool returned error → retry / replan).
4. **Tool use** — способность вызывать внешние инструменты (search, code interpreter, БД, API), расширяющие возможности LLM за пределы её весов.
5. **Memory** — короткая (контекст текущей сессии) и долгая (vector store, summary, journal) память.
6. **Planning / reasoning** — декомпозиция задачи, выбор стратегии, корректировка плана.
7. **Iterativity** — действует не одним вызовом, а в цикле, переоценивая ситуацию.

### Отличие от plain chatbot

Чатбот = одна LLM, которая отвечает на сообщение пользователя одним forward-pass. Контекст ограничен текстом, действий нет, после ответа состояние сбрасывается.

Агент:
- многошаговый цикл (а не «вход → выход»);
- доступ к внешнему миру через tools;
- состояние / память между шагами;
- собственная инициатива в выборе действий;
- цель, к которой он движется, может занимать минуты/часы и десятки шагов.

### Где появилось понятие

- В классике AI «agent» — стандартная абстракция (Russell & Norvig, AIMA);
- Первый широко известный современный LLM-агент — **ReAct** (Yao et al., 2022): «Reasoning + Acting», цикл Thought-Action-Observation;
- **AutoGPT / BabyAGI** (2023) — open-source агенты на GPT-4, популяризовали идею;
- **Voyager** (Wang et al., 2023) — embodied LLM-агент в Minecraft с skill-acquisition;
- **OpenAI Function calling / Anthropic Tool Use** — стандартизация tool-вызовов на уровне API;
- **Computer Use** (Anthropic, 2024) — агент, управляющий мышью/клавиатурой компьютера;
- **Multi-agent systems** (AutoGen, CrewAI) — коллективы агентов.

### Спектр от plain LLM до автономного агента

В сообществе условно выделяют уровни агентности:

1. **Direct Q&A** — одноразовый вопрос-ответ;
2. **Tool-augmented LLM** — LLM с фиксированным набором tools, один-два вызова;
3. **ReAct-агент** — цикл thought-action-observation, пока цель не достигнута;
4. **Planner-Executor** — декомпозиция плана + последовательное исполнение;
5. **Autonomous agent** — длинные цели, самокоррекция, скиллы, долгая память;
6. **Multi-agent systems** — несколько агентов взаимодействуют между собой.

### Проблемы и ограничения

- **Накопление ошибок** на длинных траекториях (compounding errors);
- **Hallucinations** в reasoning-шагах;
- **Стоимость** — много LLM-вызовов на одну задачу;
- **Безопасность / alignment** — агент с автономией и tool-доступом может нанести реальный вред (отправка письма, удаление файлов, расход денег);
- **Оценка** — стандартных бенчмарков пока меньше, чем для базовых LLM (AgentBench, GAIA, SWE-bench, WebArena).

## Q2. (IA)³ и адаптерные методы (A³ / Adapter family)

**(IA)³** — Infused Adapter by Inhibiting and Amplifying Inner Activations (Liu et al., 2022, статья «Few-Shot Parameter-Efficient Fine-Tuning Is Better and Cheaper than In-Context Learning»). Это **PEFT**-метод (parameter-efficient fine-tuning): дообучение очень небольшого числа параметров при замороженной базовой модели.

### Идея (IA)³

Внутри каждого трансформер-слоя добавляются три **векторных** обучаемых параметра, которые **поэлементно масштабируют** активации в трёх местах:

- $l_k$ — масштаб для **Key**-проекции в self-attention;
- $l_v$ — масштаб для **Value**-проекции в self-attention;
- $l_{ff}$ — масштаб для активации в FFN после первой Linear (перед нелинейностью).

Формально, если $K = X W_K$, то модифицированный $K' = (l_k \odot K)$, где $\odot$ — поэлементное произведение, и $l_k$ — обучаемый вектор размерности $d_k$. Аналогично для $V$ и FFN-активации.

То есть:
- (IA)³ **не добавляет матриц**, как LoRA, а только обучаемые **векторы** (диагональные scaling-матрицы), которые усиливают (>1) или подавляют (<1) определённые каналы активаций;
- Параметров получается **в десятки раз меньше**, чем у LoRA: для T0-3B всего ~0.01% от полной модели;
- При этом качество в few-shot fine-tuning сравнимо или лучше LoRA.

### Сравнение с LoRA

| Свойство | LoRA | (IA)³ |
|---|---|---|
| Что добавляется | две матрицы $A \in \mathbb{R}^{r \times d}, B \in \mathbb{R}^{d \times r}$ | три вектора $l_k, l_v, l_{ff}$ |
| Где применяется | обычно $W_q, W_v$ (или больше) | $K, V, \text{FFN}$ |
| Тип операции | $W + BA$ (rank-r addition) | $\odot l$ (element-wise scaling) |
| Параметры (T0-3B) | ~0.3% | ~0.01% |
| Inference overhead | можно слить в $W$ — нулевой | можно слить — нулевой |
| Качество | очень хорошее | сопоставимое в few-shot |

### Преимущества

1. **Минимальное число параметров** — самый «лёгкий» PEFT-метод;
2. **Можно слить** в базовые веса перед инференсом ($W_K \to W_K \cdot \mathrm{diag}(l_k)$) — нулевой overhead;
3. **Хорошо для few-shot** — превосходит in-context learning при сравнимой стоимости;
4. **Множество (IA)³-задач** легко переключать как «patches».

### Недостатки

- На больших задачах LoRA / full FT обычно слегка лучше;
- Менее популярен в продакшене, чем LoRA.

### A³ и семейство адаптерных методов

В формулировке билета упомянуто «A³». Чёткого канонического метода под этим названием нет, поэтому в широком смысле под «(IA)³, A³» обычно понимают семейство **адаптерных PEFT-методов**:

#### Houlsby Adapter (Houlsby et al., 2019)

Внутри трансформер-блока добавляется маленький **bottleneck-MLP**:

$$h \leftarrow h + W_{\text{up}}\, \sigma(W_{\text{down}}\, h),$$

с downsample $d \to r$ и upsample $r \to d$. Базовая модель заморожена, обучаются только адаптеры. Добавляют по 2 адаптера в слой (после attention и после FFN).

Минус: вносит **дополнительные параметры в forward-pass** на инференсе (нельзя слить).

#### AdapterFusion

Несколько адаптеров под разные задачи комбинируются на этапе fine-tune через attention-механизм между ними.

#### Compacter (Mahabadi et al., 2021)

Адаптер с low-rank Kronecker-product-параметризацией — ещё меньше параметров, чем Houlsby.

#### Prefix-Tuning (Li & Liang, 2021)

Добавляет обучаемые **prefix-векторы** в KV-кэш каждого слоя. Базовая модель и веса заморожены, обучается только prefix.

#### Prompt-Tuning (Lester et al., 2021)

Самый минималистичный: обучаемый soft prompt **только на входе** (несколько embedding-векторов). Качество растёт с размером модели.

#### P-Tuning v2

Соединение Prefix-Tuning с двунаправленными моделями.

#### LoRA / QLoRA

См. билет 19. Доминирующий PEFT в 2023+.

### Когда что выбирать

- **LoRA / QLoRA** — самый универсальный, продакшен-стандарт;
- **(IA)³** — когда нужно много задач с минимальным числом параметров на каждую;
- **Prefix / Prompt-tuning** — для очень больших моделей, где даже LoRA дорог;
- **Houlsby Adapter** — устаревает, но всё ещё используется в multilingual setups (AdapterHub).

### Общая идея PEFT (для понимания контекста)

Полный fine-tune миллиардной модели — миллиарды параметров, десятки GB градиентов и оптимизатора. PEFT обучает 0.01–1% параметров → значительно меньше памяти, дешевле, легко переключать «patches» под разные задачи без хранения копий полной модели. Качество — близко к full fine-tune на большинстве downstream-задач.
