# Билет 6

## Q1. Main components of an LLM agent

**LLM-агент** — это система, в центре которой стоит LLM как «мозг» принятия решений, но вокруг неё построена обвязка из памяти, инструментов и циклов планирования/исполнения, превращающая LLM из чат-собеседника в автономную сущность, способную решать задачи через действия в среде. Стандартная декомпозиция выделяет следующие ключевые компоненты.

### 1. LLM-ядро (Brain / Reasoning module)

Сама большая языковая модель (как правило, instruction-tuned + RLHF/DPO). Отвечает за:
- понимание запроса пользователя;
- декомпозицию задачи;
- выбор инструмента;
- генерацию итогового ответа;
- генерацию текста/кода как самостоятельного действия.

Современные агенты часто используют разные LLM на разных шагах: дешёвые/быстрые на routing, мощные на reasoning.

### 2. Память (Memory)

Память агента обычно делят на два уровня:

**Короткая память (short-term, working memory)** — это окно контекста LLM: текущая история диалога, scratchpad, наблюдения от инструментов. Ограничена размером контекста модели. Часто используется специальный буфер последних $k$ сообщений + summary прошлого.

**Долгая память (long-term memory)** — внешнее хранилище:
- **Vector store** (FAISS, Chroma, Pinecone) для эмбеддингов прошлых эпизодов / документов;
- **Knowledge base** или **knowledge graph** со структурированными фактами;
- **Файловая система / DB** для persistent state;
- **Episodic memory** — журнал прошлых задач и их результатов.

Агент обращается к долгой памяти через retrieval (RAG-стиль) — вытаскивает только релевантное под текущий запрос.

### 3. Планировщик (Planner)

Модуль, разбивающий задачу на шаги. Подробнее см. [билет 18](#/18). Реализации — от просто промпта «сделай план» до Tree-of-Thoughts и Plan-and-Execute. Может перепланировать на основе наблюдений (Reflexion, CRITIC).

### 4. Инструменты (Tools / Skills)

Набор внешних функций, которые агент может вызвать. Каждый tool описывается:
- именем;
- описанием на естественном языке (LLM выбирает по нему);
- JSON-schema аргументов;
- side-effects.

Типичные инструменты: web search, code interpreter, calculator, file I/O, vector search, API-клиенты, OS-команды, vision, voice, robotic actions. Подключение часто стандартизуется через **MCP** (Model Context Protocol), function calling в OpenAI/Anthropic API.

**Skills** — более высокоуровневая абстракция: преднабранные процедуры (например, «сгенерируй PDF», «отправь email»), которые агент может переиспользовать (см. [билет 11](#/11)).

### 5. Исполнитель (Executor)

Компонент, физически вызывающий tool по решению LLM. Обрабатывает ошибки, тайм-ауты, форматирует результат для возврата в LLM как `Observation`. В простых агентах сливается с Orchestrator'ом.

### 6. Reflection / Self-critique module

Дополнительный шаг, на котором агент **оценивает собственную работу**. Reflexion (Shinn et al., 2023): после каждого эпизода агент пишет в память «что пошло не так», и в следующих попытках использует этот вывод. CRITIC: агент проверяет свой ответ через инструменты (search, code execution) и исправляет ошибки.

### 7. Окружение (Environment)

Всё внешнее по отношению к агенту: пользователь, веб, файловая система, ОС, другие агенты, физический мир (для embodied). Предоставляет агенту наблюдения (observations) в ответ на действия.

### Типовой цикл

```
User goal
   │
   ▼
LLM (Brain) ──► Plan
   │
   ▼
для каждого шага:
   LLM решает → Action (tool call) → Executor → Observation
   ▼
   Обновление памяти → Reflection (опц.) → Re-plan (опц.)
   │
   ▼
Final answer
```

Это и есть **Thought-Action-Observation** loop (ReAct, см. [билет 16](#/16)).

### Взаимосвязь компонентов

Без памяти агент забывает прошлые эпизоды; без планировщика хаотично пытается; без инструментов ограничен «знаниями в весах»; без рефлексии повторяет ошибки; без оркестрации (см. [билет 5](#/05)) компоненты не складываются в систему.

## Q2. GPT: Architecture, Training

**GPT** (Generative Pre-trained Transformer; Radford et al., OpenAI, 2018) — decoder-only авторегрессивный трансформер. Здесь — детальный разбор архитектуры и тренировки.

### Архитектура decoder-only

GPT — стек из $L$ одинаковых трансформер-блоков:

```
input → Token Emb + Position Emb
              │
       ┌──────▼──────┐
       │             │
       │  ┌───────┐  │
       │  │ LN    │  │
       │  │ MSA   │ ← causal mask
       │  │ +res  │  │
       │  └───────┘  │
       │  ┌───────┐  │
       │  │ LN    │  │
       │  │ FFN   │  │
       │  │ +res  │  │
       │  └───────┘  │
       └──────┬──────┘  × L
              │
        Final LN
              │
        Linear → Vocab → softmax
```

**Ключевая деталь — causal mask** в self-attention. Маска $M_{ij} = -\infty$ при $j > i$, иначе 0:

$$\mathrm{Attn}(Q, K, V) = \mathrm{softmax}\!\left(\frac{QK^\top}{\sqrt{d_k}} + M\right) V.$$

Гарантирует, что токен видит только себя и левых соседей, что соответствует авторегрессивной природе LM.

### Компоненты блока

1. **Multi-Head Self-Attention (MSA)**: $H$ голов, каждая с $W_Q^{(h)}, W_K^{(h)}, W_V^{(h)}$, параллельные attention'ы конкатенируются и проецируются $W_O$.
2. **FFN** (position-wise): обычно $\mathrm{FFN}(x) = \sigma(xW_1)W_2$, размерность $4d$ (в современных LLaMA — SwiGLU).
3. **LayerNorm** — pre-LN в GPT-2+ (стабильнее), post-LN в GPT-1.
4. **Residual connections** вокруг MSA и FFN.

### Размеры (на примере GPT-3 175B)

- $L = 96$, $d = 12288$, $H = 96$, $d_h = 128$;
- Контекст — 2048 токенов (GPT-3), позже расширяется;
- Параметры словаря (BPE-tokenizer): ~50K;
- Всего параметров: 175B.

### Цель претрейнинга — Causal Language Modeling

Дан корпус токенов $X = (x_1, x_2, \ldots, x_T)$. Минимизируется отрицательное log-правдоподобие:

$$\mathcal{L}_{\mathrm{LM}}(\theta) = -\sum_{t=1}^{T} \log P_{\theta}\!\left(x_t \mid x_{1:t-1}\right).$$

Это **single-task** объект: всё, что модель учит, — предсказывать следующий токен. Никакого MLM, никакого NSP. Эта простота и масштабируемость на огромные корпуса — ключ успеха.

### Данные

GPT-3: ~300B токенов из mix:
- **Common Crawl** (filtered, 410B raw → ~410B used);
- **WebText2** (45B);
- **Books1, Books2** (~67B);
- **Wikipedia** (~3B).

Дедупликация, фильтрация качества, балансировка между источниками — критично для качества.

### Tokenizer

**BPE** (Byte-Pair Encoding) или его варианты (Tiktoken). Словарь обычно 30K–100K. Современные LLM используют byte-level BPE, чтобы покрыть любые символы.

### Тренировочные детали

- **Optimizer**: AdamW с $\beta_1 = 0.9$, $\beta_2 = 0.95$, weight decay 0.1;
- **LR schedule**: warmup + cosine decay;
- **Mixed precision**: BF16 / FP16;
- **Distributed training**: data parallel + tensor parallel + pipeline parallel + ZeRO/FSDP;
- **Batch size**: миллионы токенов на шаг;
- **Compute**: десятки тысяч GPU-часов на обучение GPT-3 (>$10^{23}$ FLOPs).

### Пост-тренировка

Современный полный пайплайн:
1. **Pretraining** — causal LM на огромном корпусе;
2. **SFT (Instruction tuning)** — дообучение на парах (instruction, response), ~10–100K пар;
3. **Alignment**: RLHF (RM + PPO) или DPO/GRPO — на парах предпочтений;
4. **Safety / red-teaming** fine-tunes;
5. **Tool-use / function-calling** доводка.

### Эмерджентные способности

С масштабом (параметры × данные × compute) у GPT возникают качественно новые свойства, которых не было у меньших моделей:

- **In-context learning** — обучение на примерах в промпте без апдейта весов;
- **Chain-of-Thought reasoning** — пошаговое рассуждение;
- **Code generation, math, переводы** — без специального fine-tuning;
- **Instruction following** — после SFT/RLHF.

Это эмпирически связано со scaling laws (см. [билет 15](#/15)) и одна из главных причин, почему индустрия движется в сторону всё больших моделей.

### Эволюция

GPT-1 (117M) → GPT-2 (1.5B) → GPT-3 (175B) → InstructGPT/ChatGPT (RLHF) → GPT-4 (предположительно MoE, мультимодальность) → GPT-4o/o1/o3/GPT-5 (reasoning через скрытый CoT, тест-time compute, мультимодальность).
