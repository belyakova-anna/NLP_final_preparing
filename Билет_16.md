## Q1. Thought-Action-Observation workflow для LLM-агентов (ReAct)

**ReAct** (Reasoning + Acting; Yao et al., ICLR 2023, «ReAct: Synergizing Reasoning and Acting in Language Models») — это парадигма промптинга и архитектуры LLM-агентов, в которой генерация чередует **рассуждения (Thought)** и **действия (Action)**, причём после каждого действия среда возвращает **наблюдение (Observation)**, которое попадает обратно в контекст. Цикл повторяется, пока агент не достигнет цели.

### Основной цикл

```
Thought_1 → Action_1 → Observation_1
Thought_2 → Action_2 → Observation_2
...
Thought_n → Final Answer
```

На каждом шаге $t$:
1. **Thought** — LLM сама себе пишет, что нужно сделать дальше («Чтобы ответить на этот вопрос, мне надо найти Х»);
2. **Action** — структурированный вызов инструмента (`Search[query]`, `Calculator[expr]`, JSON function call);
3. **Observation** — результат вызова, возвращённый средой (text, JSON, error message);
4. Эти три записи добавляются к контексту, и цикл идёт дальше до тех пор, пока модель не сгенерирует `Final Answer` или не упрётся в лимит шагов.

### Пример (из оригинальной статьи)

Вопрос: «Apart from the Apple Remote, what other device can control the program Apple Remote was originally designed to interact with?»

```
Thought 1: I need to find what program the Apple Remote was originally designed to interact with.
Action 1: Search[Apple Remote]
Observation 1: The Apple Remote is a remote control device... originally designed to control Front Row.
Thought 2: I need to search Front Row to find what other devices can control it.
Action 2: Search[Front Row (software)]
Observation 2: Front Row is discontinued... It can be controlled by Apple Remote or keyboard function keys.
Thought 3: So the answer is keyboard function keys.
Action 3: Finish[keyboard function keys]
```

### Формат промпта

ReAct реализуется обычно через **few-shot prompting**: в системный промпт добавляются 2–6 решённых примеров в формате `Thought / Action / Observation`. LLM обучается на лету подражать формату. После последнего in-context-примера ставится текущий запрос — модель продолжает в том же стиле.

В современных API (OpenAI function calling, Anthropic tool use) ReAct реализован «прозрачнее»: модель генерирует action как структурированный JSON-вызов, оркестратор вызывает tool, наблюдение приходит как `tool_result` сообщение. Thought-часть остаётся в виде текстового ответа модели или скрытого reasoning.

### Зачем нужны и Thought, и Action

Главное концептуальное достижение ReAct — **синергия reasoning и acting**:

- **Только CoT (без действий)**: модель «рассуждает в вакууме», не имея доступа к свежей или точной информации — отсюда галлюцинации в фактических вопросах.
- **Только actions (без thought)**: модель быстро превращается в reflex-агента, теряет долгосрочный план, действует наобум.
- **ReAct**: thought обосновывает выбор действия, action обогащает контекст реальной информацией, в сумме система делает меньше ошибок и легче восстанавливается после неудач.

В оригинальной статье на HotPotQA (multi-hop QA) и Fever (factchecking) ReAct превзошёл и чистый CoT, и chain-of-tool-use.

### Преимущества vs CoT

| | CoT | ReAct |
|---|---|---|
| Доступ к актуальной информации | нет | есть (через tools) |
| Проверка фактов | модель «уверенно ошибается» | через search/db |
| Длинные многошаговые задачи | склонна к hallucination | пошаговая верификация |
| Цена | дешевле | дороже (несколько LLM-вызовов) |
| Реализация | один промпт | требует tool-runtime |

### Ограничения ReAct

1. **Длина контекста**: на каждом шаге Thought + Action + Observation добавляются в контекст, и тот быстро растёт; помогают memory-management, summarization, sliding window;
2. **Compounding errors**: ошибочное Thought → ошибочное Action → бесполезное Observation → дальнейшие шаги ухудшаются. Reflexion / self-critique частично спасают;
3. **Стоимость**: один ReAct-эпизод может занимать 5–50 LLM-вызовов;
4. **Непредсказуемая остановка**: модель может зациклиться, если нет жёсткого лимита шагов или критерия завершения;
5. **Tool errors**: ReAct рассчитывает на корректные observation; если tool возвращает шум, модель может «галлюцинировать» интерпретацию.

### Расширения

- **Reflexion** (Shinn et al., 2023) — после неудачного эпизода LLM пишет «урок» в long-term memory; следующая попытка стартует с этим уроком в контексте;
- **Plan-and-Execute** — Planner один раз пишет план (без observation), Executor шагает по нему ReAct-стилем;
- **Tree-of-Thoughts** — вместо линейного цикла строится дерево thought-веток с поиском;
- **CodeAct** (Wang et al., 2024) — Action всегда Python-код, который может вызвать любые tools внутри; унифицирует action space.

### Связь с курсом

ReAct — самая базовая и при этом самая важная парадигма агента: на ней построен большинство практических LLM-агентов (LangChain Agents, AutoGPT, BabyAGI, Voyager, Anthropic Computer Use). Это конкретная имплементация Thought-Action-Observation workflow и стандартный ответ на вопрос «как сделать tool-using агента».

## Q2. Sampling. Beam-search, Top-K, Top-P. Temperature

**Sampling** — это процесс выбора следующего токена из распределения, выданного LLM на каждом шаге авторегрессивной генерации:

$$P(x_t \mid x_{1:t-1}) = \mathrm{softmax}(z_t),$$

где $z_t \in \mathbb{R}^{|V|}$ — логиты по словарю. Выбор стратегии существенно влияет на качество и разнообразие генерации.

### Greedy decoding

Самая простая стратегия: на каждом шаге берём $\arg\max P$:

$$x_t = \arg\max_{v \in V} P(v \mid x_{1:t-1}).$$

- Детерминированно;
- Часто приводит к локально-оптимальной, но глобально-плохой последовательности (повторы, скучные ответы);
- Использует mode распределения, но игнорирует более длинные пути с большим joint-likelihood.

### Beam Search

Поддерживает $B$ (beam width) **гипотез одновременно**. На шаге $t$ из каждой текущей гипотезы расширяется на топ-$B$ продолжений → получается $B \cdot B$ кандидатов; из них оставляется топ-$B$ по cumulative log-prob.

$$\text{score}(y_{1..t}) = \sum_{i=1}^t \log P(y_i \mid y_{1:i-1}).$$

Финальная гипотеза с лучшим скором.

#### Особенности

- **Менее «жадно»**: видит дальше на $B$ путей и не упирается в локальный максимум;
- **Длиновая нормализация**: длинные последовательности имеют меньший cumulative log-prob → используется length penalty:
$$\text{score}_{\text{norm}} = \frac{\text{score}}{((5+|y|)/6)^\alpha};$$
- **Стандарт для машинного перевода и summarization**, где нужна **точность**;
- **Плох для open-ended генерации** (диалог, истории) — даёт скучный, повторяющийся текст (исследование Holtzman et al. 2019);
- **Дорогой**: в $B$ раз больше compute и памяти.

### Sampling-based стратегии

Open-ended генерация требует **разнообразия** — пользователю интересно получить не самый вероятный токен, а правдоподобный с шумом.

#### Pure (multinomial) sampling

$$x_t \sim P(x_t \mid x_{1:t-1}).$$

Просто семплируем по полному распределению. Беда: длинный «хвост» очень малых вероятностей содержит много нерелевантных токенов; их сумма ненулевая → периодически вылезает мусор.

#### Top-K sampling

Оставляем только $K$ токенов с наибольшей вероятностью, пере-нормализуем и семплируем оттуда:

$$P_{\text{top-}K}(v) = \begin{cases} P(v) / Z, & v \in \text{top-}K \\ 0, & \text{иначе} \end{cases}, \quad Z = \sum_{v \in \text{top-}K} P(v).$$

Типичные $K$: 40, 50.

- **Минусы**: фиксированный $K$ плохо адаптируется к форме распределения. Если оно очень peaked (один токен с $P=0.95$), то $K=50$ всё равно включит мусор. Если оно flat (много токенов с похожей вероятностью), то $K=50$ может отрезать релевантные варианты.

#### Top-P (nucleus) sampling

Holtzman et al., ICLR 2020. Вместо фиксированного $K$ — **минимальный набор токенов**, чья суммарная вероятность $\ge p$:

$$\operatorname{nucleus}(p) = \min\bigl\{|S| : \sum_{v \in S} P(v) \ge p\bigr\},$$

где $S$ упорядочен по убыванию вероятности. Затем нормализуем и семплируем.

- $p = 1.0$ → pure sampling;
- $p = 0.9$ или $0.95$ — стандарт;
- Адаптивно: на peaked-распределении nucleus маленький, на flat — большой;
- **Лучшая стратегия для open-ended генерации** на момент 2020 года.

#### Min-P, Typical, η-sampling — современные альтернативы

- **Min-P** — более устойчивый аналог Top-P;
- **Typical sampling** — выбираем токены, чья энтропийная «типичность» близка к ожидаемой;
- **Locally typical sampling, η-sampling, Mirostat** — разные варианты адаптивных стратегий.

### Temperature

**Temperature $T$** — параметр, который сглаживает или заостряет распределение перед семплингом. Логиты $z$ делятся на $T$ перед softmax:

$$P_T(v) = \mathrm{softmax}(z / T).$$

- $T \to 0$: распределение → точечная масса в argmax → greedy;
- $T = 1$: оригинальное распределение модели;
- $T > 1$: распределение более **плоское** → больше разнообразия, выше шанс «творческих»/неверных токенов;
- $T < 1$: более **острое** → ближе к greedy, безопаснее, но скучнее.

Типичные значения:
- **Code, math, factual QA**: $T \in \lbrack 0,\,0.3\rbrack$ — точность важнее разнообразия;
- **Чат, обычные задачи**: $T \in \lbrack 0.5,\,0.8\rbrack$;
- **Creative writing, stories, brainstorming**: $T \in \lbrack 0.8,\,1.2\rbrack$.

Temperature обычно комбинируется с Top-K или Top-P (сначала фильтр, потом семпл).

### Repetition penalty / no-repeat-n-gram / presence penalty

Дополнительные приёмы:

- **Repetition penalty** (Keskar et al., CTRL): уменьшает вероятность ранее появившихся токенов;
- **No-repeat-n-gram**: запрещает повторять n-граммы;
- **Frequency / presence penalty** (OpenAI API): аналогичные регуляризации против повторов.

### Practical guidelines

| Задача | Рекомендация |
|---|---|
| Точная фактическая QA | $T = 0$, greedy / beam(B=5) |
| Перевод, summarization | beam(B=4–8), length penalty, no-repeat |
| Code generation | $T = 0.0$–$0.3$, top-p 0.95 |
| Чат | $T = 0.7$, top-p 0.9 |
| Creative writing | $T = 0.9$–$1.1$, top-p 0.95 |
| CoT + Self-Consistency | $T = 0.7$, top-p 0.95, $N$ samples |

### Связь с курсом

Sampling — нижний уровень всех высокоуровневых техник: in-context learning, CoT, Self-Consistency, ReAct — все они в финальной точке используют какую-то sampling-стратегию для генерации. Self-Consistency, в частности, требует $T > 0$ и работает за счёт усреднения по нескольким семплам.
