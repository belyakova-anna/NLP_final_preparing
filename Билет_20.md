# Билет 20

## Q1. LLM-as-judge

**LLM-as-a-judge** — парадигма, в которой одна LLM оценивает выходы другой LLM (или сравнивает выходы нескольких моделей) вместо человеческого асессора. Появилась как ответ на проблему: разметка людьми дорогая и медленная, классические автоматические метрики (BLEU, ROUGE, BERTScore) плохо коррелируют с человеческими предпочтениями для open-ended-генерации (диалоги, объяснения, код, reasoning). LLM-судья (GPT-4, Claude, Llama-3-70B-Instruct и т. п.) ставит оценки или сравнивает ответы, имитируя суждения человека.

### Где применяется

1. **Evaluation / бенчмарки**:
   - **MT-Bench** (Zheng et al., NeurIPS 2023) — мульти-туровые диалоги, GPT-4 оценивает каждый ответ по 10-балльной шкале;
   - **AlpacaEval / AlpacaEval 2** — pairwise winrate против эталонной модели;
   - **Arena-Hard**, **WildBench**, **MixEval** — современные leaderboard'ы;
   - **G-Eval** (Liu et al., 2023) — judge с CoT и форматом «дай оценку 1–5 по этому критерию» (faithfulness, fluency, relevance);
   - **Prometheus** — open-source judge-LLM, обученная специально под evaluation.

2. **Reward modeling / RLAIF** — судья выдаёт preference signal вместо человеческой разметки (Bai et al., Constitutional AI, 2022).

3. **Online evaluation в production** — LLM-судья мониторит качество реальных ответов системы.

4. **Self-consistency / self-verification** — модель сама оценивает свои промежуточные шаги (Reflexion, CRITIC).

### Форматы оценки

- **Pointwise / absolute scoring** — судье даётся ответ и критерии, он ставит число (например, 1–5 по relevance):
  ```
  Question: ...
  Answer: ...
  Rate the answer's helpfulness from 1 to 5: ...
  ```
- **Pairwise comparison** — судье даются два ответа A и B на один промпт, он выбирает лучший («A», «B», «tie»). Считается более надёжным, чем pointwise.
- **Reference-based** — есть эталонный ответ, судья сравнивает кандидата с ним.
- **Multi-aspect** — несколько критериев одновременно (helpful, factual, safe, fluent), каждый отдельно.

### Известные смещения (biases)

LLM-судьи систематически ошибаются. Из MT-Bench и последующих работ:

1. **Position bias** — судья предпочитает ответ, идущий первым (или иногда вторым). Митигация: запускать сравнение в обоих порядках и брать только согласованные ответы.
2. **Length bias / verbosity bias** — длинные ответы выигрывают чаще, чем заслуживают.
3. **Self-preference bias / self-enhancement** — судья предпочитает ответы своей же модели или модели той же семьи.
4. **Style bias** — формат (markdown, нумерованные списки, уверенный тон) важнее содержания.
5. **Sycophancy bias** — судья соглашается с предположениями в промпте.
6. **Limited reasoning** — судья плохо проверяет математику и сложную фактуру.

### Митигации

- **Position randomization + agreement filter**: спрашивать в обоих порядках, считать только когда оценки согласованы;
- **Chain-of-Thought prompting судьи** — заставить рассуждать прежде чем выносить оценку;
- **Reference answers** — давать судье эталон;
- **Calibrated rubrics** — детальные критерии, а не просто «оцените 1–10»;
- **Ensemble of judges** — несколько разных LLM голосуют;
- **Human spot-checks** — выборочная человеческая верификация;
- **Specialized judge models** — Prometheus, JudgeLM, Auto-J, обученные специально под оценку.

### Качество судей

GPT-4 как судья достигает ~80% согласия с экспертами на MT-Bench, что близко к согласию между двумя людьми (~80–85%). На сложных доменах (математика, сложный код, безопасность) согласие ниже. Для open-ended subjective задач GPT-4 / Claude-3 — приемлемая замена человека; для precise factuality нужен grounded eval (с reference / retrieval).

### Связь с RLAIF / Constitutional AI

В **RLAIF** LLM-судья даёт preference labels, которые идут на обучение reward-модели (вместо человеческих разметчиков):

`prompt → 2 ответа от полиси → LLM-судья выбирает лучший → пара (chosen, rejected) → RM → PPO/DPO`.

В **Constitutional AI** (Anthropic) судья также используется как Critic в self-revise loop: ответ модели проверяется на соответствие принципам «конституции», при нарушении модель сама исправляет ответ. Это позволяет масштабировать alignment без дорогих человеческих разметчиков.

## Q2. Group Relative Policy Optimization (GRPO)

**GRPO** — упрощение PPO, в котором отказались от value-сети (критика). Вместо неё advantage оценивается через **сравнение нескольких ответов на один и тот же промпт внутри группы**. Это снижает память (не нужен критик размером с полиси) и устраняет источник нестабильности (плохо обученный критик портит обучение). Метод стал известен благодаря **DeepSeek-Math** (Shao et al., 2024) и **DeepSeek-R1** (2025), где он использовался для RL по математическим и кодовым задачам с верифицируемой наградой.

### Мотивация: проблемы PPO

Стандартный PPO в RLHF требует трёх копий модели в памяти:
1. **Полиси** $\pi_\theta$ (обучаемая);
2. **Reference** $\pi_{ref}$ (замороженная, для KL-якоря);
3. **Value-сеть / critic** $V_\phi$ (обычно той же архитектуры, что полиси) — оценивает $\hat A_t$ через GAE.

Critic — отдельный источник нестабильности: его нужно тренировать одновременно с полиси, плохо обученный value завышает или занижает advantage и ломает PPO. Также критик удваивает требования к памяти и добавляет complexity.

GRPO заменяет critic на **групповую нормализацию награды**.

### Алгоритм пошагово

1. **Sampling**: для каждого промпта $x$ из батча сэмплируется группа из $G$ ответов $\{y_1, \ldots, y_G\}$ от текущей полиси $\pi_{\theta_{old}}$ (типично $G = 4{-}64$).
2. **Reward**: каждый ответ оценивается reward-функцией: $r_1, \ldots, r_G$. В DeepSeek-Math/R1 это **rule-based** награда (правильность ответа в математике, прохождение тестов в коде); может быть и обученный RM.
3. **Group-relative advantage**: награды нормализуются внутри группы:
$$\hat A_i = \frac{r_i - \operatorname{mean}(\{r_1,\ldots,r_G\})}{\operatorname{std}(\{r_1,\ldots,r_G\})}.$$
Это и есть advantage без критика — каждый ответ сравнивается со средним по своей группе. На уровне токенов часто $\hat A_i$ просто копируется на все токены ответа $y_i$ (либо распределяется иначе).
4. **PPO-clipped objective** (тот же, что в стандартном PPO):
$$L^{GRPO}(\theta) = \mathbb{E}_{x, \{y_i\}} \frac{1}{G} \sum_{i=1}^G \min\Bigl(r_t(\theta) \hat A_i,\ \mathrm{clip}(r_t(\theta), 1-\varepsilon, 1+\varepsilon)\, \hat A_i\Bigr),$$
где $r_t(\theta) = \pi_\theta(a_t \mid s_t) / \pi_{\theta_{old}}(a_t \mid s_t)$.
5. **KL-якорь**: добавляется штраф к референсной (SFT) модели — как в PPO-RLHF:
$$L^{GRPO}_{total} = L^{GRPO} - \beta\, \mathrm{KL}[\pi_\theta \| \pi_{ref}].$$
В DeepSeek используется **unbiased KL estimator** $\hat k_3 = \pi_{ref}/\pi_\theta - \log(\pi_{ref}/\pi_\theta) - 1 \ge 0$ — он всегда положительный, в отличие от обычной выборочной оценки $\log(\pi_\theta / \pi_{ref})$.

### Особенности

- **Нет value-сети** — экономия ~1/3 памяти, +простота;
- **Стабильность**: групповая нормализация даёт хорошую оценку advantage, не смещённую плохим критиком;
- **Подходит для reasoning-задач**: можно сгенерировать много кандидатов на промпт и сравнить — органично для математики/кода;
- **Rule-based reward** не подвержен reward hacking так же, как обученный RM (нельзя «обмануть» автоматический проверщик примера, нужно реально решить задачу).

### Гиперпараметры (типично)

- $G \approx 16{-}64$ (DeepSeek-R1 использовала $G=64$);
- $\varepsilon = 0.2$ (как в PPO);
- $\beta \in \lbrack 10^{-3},\, 10^{-2}\rbrack$ для KL.

### Сравнение с PPO и DPO

| | PPO | DPO | GRPO |
|---|---|---|---|
| Reward model | да | имплицитно | да (rule-based или RM) |
| Critic / value | да | нет | **нет** |
| Online sampling | да | нет | да |
| Память | большая | малая | средняя |
| Стабильность | средняя | высокая | высокая |
| Подходит для | RLHF | offline preference | reasoning / verifiable rewards |

### Использование в DeepSeek-R1 и emerging reasoning

GRPO стал ключевым кирпичом «reasoning»-LLM 2024–2026:
- **DeepSeek-Math** (2024) — GRPO + verifiable math rewards → SOTA на MATH;
- **DeepSeek-R1** (2025) — GRPO с rule-based наградами на математических и code-задачах, цепочки рассуждений (CoT) удлиняются автоматически в процессе RL, эмерджентные способности «aha-moment»;
- **Kimi-K1.5, QwQ, OpenAI-o1/o3** — родственные подходы (точные алгоритмы o1/o3 не раскрыты, но идея инференс-time-CoT + RL по верифицируемым наградам общая).

GRPO показал, что для задач с верифицируемой наградой можно делать масштабный RL без human labelling и без отдельной value-сети, получая сильное улучшение reasoning-способностей у LLM.
