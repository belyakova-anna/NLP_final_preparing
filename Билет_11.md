# Билет 11

## Q1. Role of skills in agentic systems

**Skill** в LLM-агенте — это **переиспользуемая, описанная процедура** для решения определённого класса задач: набор инструкций, инструментов, шагов и метаданных. Скилл хранится в **skill library** и подгружается агентом по запросу. В отличие от обычного tool-call (атомарного действия), скилл — это многошаговая стратегия, чаще всего более высокого уровня абстракции.

### Что входит в скилл

Типичный скилл — это пакет, состоящий из:

- **Имени и описания** (для retrieval-а: «когда применять»);
- **Промпта / инструкций** (как именно решать задачу пошагово);
- **Списка нужных инструментов / API**;
- **Примеров** (few-shot демонстраций);
- **Метаданных**: входы, выходы, требования (напр. нужен интернет, нужен code interpreter), ограничения;
- Иногда **кода** (Python-функций, скриптов), который агент исполняет.

Например, скилл «составь pptx-презентацию» содержит инструкции, перечень шаблонов, ссылки на python-pptx, примеры — агент при выборе этого скилла сразу получает «готовый рецепт» и не пытается изобретать решение с нуля.

### Зачем скиллы

1. **Переиспользование** — один раз отлаженная процедура работает на тысячах запросов;
2. **Декомпозиция знаний** — общая LLM остаётся универсальной, специализация выносится в скиллы;
3. **Ограничение пространства поиска** — вместо «придумать с нуля» агент сразу знает workflow;
4. **Композируемость** — сложная задача собирается из нескольких скиллов;
5. **Сменяемость** — обновили скилл → агент сразу использует новую версию (без перетренировки модели);
6. **Аудит и контроль** — поведение скилла легко тестировать и ревьюить, в отличие от свободной генерации.

### Selection / retrieval скилла

Когда скиллов много, нужен механизм выбора:

- **Description-based retrieval**: эмбеддинги названий и описаний скиллов, semantic search по запросу пользователя — выбираются top-k релевантных скиллов;
- **Tool-RAG**: тот же RAG-подход, только индексируются скиллы / tools, не документы;
- **Routing-LLM**: отдельная (часто более лёгкая) LLM-классификатор решает, какой скилл подходит;
- **Hierarchical routing**: сначала категория (research / coding / writing), потом конкретный скилл внутри.

### Skill acquisition (обучение скиллов)

- **Вручную написанные** — типичный путь в production-агентах (Claude Skills, Anthropic; OpenAI GPTs; LangChain templates);
- **Автоматически приобретаемые** — самый известный пример: **Voyager** (Wang et al., 2023) в Minecraft. Агент решает задачу, проверяет результат через игру, при успехе **запоминает решение как скилл** в библиотеке. Со временем библиотека растёт, агент решает всё более сложные задачи композицией старых скиллов.
- **Из демонстраций** — агент наблюдает, как человек решает задачу, и формализует это в скилл.

### Voyager как канонический пример

В Voyager агент в Minecraft:
1. **Curriculum**: GPT-4 предлагает следующую задачу (от простого к сложному);
2. **Skill library**: хранит уже отлаженные функции на JavaScript;
3. **Iterative prompting**: пишет код для текущей задачи, запускает, при ошибке — анализирует и переписывает;
4. **Skill addition**: успешный код добавляется в библиотеку с описанием.

После обучения агент решал намного больше задач, чем без skill library, потому что мог комбинировать ранее освоенные навыки.

### Skills в современных продакшен-системах

- **Anthropic Claude Skills** — папки с `SKILL.md`, инструкциями и опциональным кодом, подгружаемые в агент по описанию;
- **OpenAI Custom GPTs / Assistants** — пользовательские скиллы с tools и instructions;
- **LangChain templates / agents marketplace**;
- **Agent Tool Repositories** в Multi-Agent системах (CrewAI, AutoGen).

### Почему скиллы важнее «голых» tools

- Tool — атомарное действие («сделай поиск», «выполни код»);
- Skill — стратегия с tool'ами + знанием о том, **как** их применять для решения класса задач.

LLM сильна в reasoning, но слаба в воспроизводимом следовании сложным процессам. Скиллы кодируют эти процессы и компенсируют слабое место.

## Q2. LLaMA architecture

**LLaMA** (Large Language Model Meta AI; Touvron et al., 2023) — серия открытых decoder-only LLM от Meta. Версии: LLaMA-1 (Feb 2023), LLaMA-2 (Jul 2023), LLaMA-3 (Apr 2024), LLaMA-3.1/3.2/3.3 (2024). Стала эталонной архитектурой современных открытых LLM — Mistral, Qwen, DeepSeek, Yi, Falcon в значительной степени её повторяют.

### Базовая идея

Decoder-only трансформер с causal language modeling, как GPT, но с рядом модернизаций по сравнению с ванильным «Attention Is All You Need» (Vaswani 2017). Каждый слой — `Self-Attention → FFN`, residual + Norm, но конкретные блоки заменены.

### Размеры

- **LLaMA-1**: 7B / 13B / 33B / 65B (1.4T токенов);
- **LLaMA-2**: 7B / 13B / 70B (2T токенов; +chat-версии после RLHF);
- **LLaMA-3**: 8B / 70B / 405B (15T токенов; tokenizer 128K vs 32K раньше);
- **LLaMA-4** (2025): MoE с активными ~17B параметрами.

### Ключевые архитектурные особенности

#### 1. Pre-Normalization с RMSNorm

Вместо post-norm (как в оригинальном трансформере) — **pre-norm**: норма применяется **до** sub-layer'а. Это стабилизирует обучение глубоких сетей.

Вместо LayerNorm — **RMSNorm** (Root Mean Square LayerNorm; Zhang & Sennrich, 2019):

$$\mathrm{RMSNorm}(x) = \frac{x}{\sqrt{\frac{1}{d}\sum_i x_i^2 + \varepsilon}} \cdot g,$$

где $g$ — обучаемый scale-параметр. В отличие от LayerNorm, нет вычисления mean — только rescaling по RMS, и нет bias. Это:
- Дешевле по compute;
- Стабильно;
- Без ухудшения качества.

#### 2. SwiGLU вместо ReLU/GELU в FFN

Стандартный FFN: `Linear → GELU → Linear`. В LLaMA — **SwiGLU** (Shazeer, 2020):

$$\mathrm{SwiGLU}(x) = (\mathrm{Swish}(x W_1) \odot x W_3)\, W_2,$$

где $\mathrm{Swish}(x) = x \cdot \sigma(x)$ — гладкая активация, $\odot$ — поэлементное умножение, $W_3$ — дополнительная gating-проекция. SwiGLU стабильно даёт прирост качества при той же compute, поэтому стал стандартом.

Размерность hidden $\frac{8}{3} d$ (вместо $4d$ как обычно), чтобы compensate за лишнюю проекцию и сохранить общее число параметров.

#### 3. Rotary Positional Embeddings (RoPE)

Никаких обучаемых positional embeddings — RoPE применяется к Q, K в каждом слое (см. билет 3 / 15). Даёт относительную позицию, экстраполяцию на длинные контексты, отсутствие дополнительных параметров.

В LLaMA-3 контекст вырос до **128K** через scaled-RoPE / NTK-aware / YaRN.

#### 4. Grouped-Query Attention (GQA) — начиная с LLaMA-2 70B и всей LLaMA-3

Для экономии KV-кэша: $G$ KV-голов на $H$ Q-голов (см. билет 4). Например, LLaMA-3 70B: 64 Q-головы, 8 KV-групп.

#### 5. Tokenizer

- LLaMA-1/2: SentencePiece BPE, 32K словарь;
- LLaMA-3: tiktoken-style BPE, 128K словарь — лучше токенизирует non-English и code, эффективнее по числу токенов.

#### 6. Без bias в линейных слоях

Линейные проекции в attention и FFN — **без bias** (как у PaLM). Экономит параметры, не вредит качеству.

### Тренировка

- **Causal language modeling**: $\mathcal{L} = -\sum_t \log P_\theta(x_t \mid x_{1:t-1})$;
- AdamW, $\beta_1=0.9, \beta_2=0.95$, weight decay 0.1, gradient clipping 1.0, cosine schedule с warmup;
- BF16 / FP8 (для LLaMA-3);
- Хорошо отфильтрованные данные: CommonCrawl + Github + arXiv + Wikipedia + Books.

### Post-training

- **SFT** на ~10K–100K инструкций;
- **RLHF** (LLaMA-2-Chat) или **DPO** (популярно в open-community);
- В LLaMA-3 — комбинация SFT + Rejection Sampling + DPO + PPO.

### Сравнительные особенности

| Компонент | Vanilla Transformer (2017) | LLaMA |
|---|---|---|
| Норма | LayerNorm post-norm | RMSNorm pre-norm |
| FFN | ReLU/GELU, $4d$ | SwiGLU, $\sim\!\frac{8}{3}d$ |
| Positional | синусоидальная absolute | RoPE relative |
| Attention | MHA | MHA / GQA |
| Bias в Linear | да | нет |
| Tokenizer | BPE | SentencePiece / tiktoken |

### Влияние

LLaMA задала стандарт «современного recipe» decoder-only LLM. Mistral, Qwen, DeepSeek, Yi, Gemma, Phi-3 — все это почти-LLaMA с теми же RoPE / RMSNorm / SwiGLU / GQA. Открытые веса LLaMA-2/3 запустили волну open-source-LLM, fine-tunes (Vicuna, WizardLM, Zephyr, Hermes), и легли в основу инфраструктуры (llama.cpp, vLLM, Ollama).
